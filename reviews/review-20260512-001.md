## 3. Questions

**Question: case 29 expected output diverges from spec §4 table.** The
spec's §4 "Palindrome rules" table (line 175) and the testing-requirements
section (line 319) both state that NFD "é" → `False`. The test file asserts
`True` and includes a comment claiming this was a spec correction issued by
the architect during Mode 2 compliance check.

I verified independently against the spec's §4 "Behavior" algorithm (lines
152-161): apply `casefold`, filter via `c.isalnum()`, compare to reverse.
Python's `str.isalnum()` returns `False` for combining marks in Unicode
category Mn, so U+0301 (combining acute) is filtered out, leaving `"e"` —
a trivial palindrome. So under the algorithm the spec actually specified,
NFD "é" returns `True`. The spec's expected output of `False` would only
be reachable if either (a) the filter used a different predicate
(e.g., `c.isalnum() or unicodedata.category(c) == 'Mn'`), or (b) NFD
normalization were applied before filtering — neither of which the spec
asks for.

So either: (a) the spec's §4 table and case 29 expectation were
internally inconsistent with the §4 algorithm, and the architect
correctly resolved the inconsistency in favor of the algorithm and the
spec's "no `unicodedata`" rule (§4 dependencies + §5 default #4), OR
(b) the architect or engineer should have flagged the inconsistency to
the user before resolving it, since the user's spec-approval was based
on §4's stated expected outputs, not on the algorithm's actual semantics.

The implementation as it stands is correct under the algorithm. I read
this as a Question rather than Blocking because:
- The behavior is internally consistent and the spec text §5 default #4
  ("No NFC / NFD normalization") explicitly anticipates this edge case
  and tells the engineer not to add normalization to fix it;
- The test ID was honestly renamed (`nfd-e-acute-filters-to-e`) so the
  divergence is not hidden;
- A user reviewing the diff will see the test ID and the inline comment
  and can object if they want the original §4 expectation.

What I would do under each interpretation:
- If architect intentionally corrected the spec: **approve** as-is. The
  algorithm-driven behavior is what the spec's §4 §5 actually requested,
  and the test ID rename is honest.
- If the spec correction should have been escalated to user first:
  surface it in Mode 3 coordination as a "user-decision item" and let
  the user choose between (i) accept the new `True` expectation, (ii)
  add `unicodedata.normalize("NFC", folded)` to make NFD "é" → `True`
  via normalization rather than filter-drop (semantically nicer), or
  (iii) change the filter to keep combining marks and revert to the
  original §4 `False` expectation. Architect's call.

This question is for the architect's Mode 3 triage, not a finding against
the engineer.
