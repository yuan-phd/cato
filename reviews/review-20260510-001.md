# Review — `slugify(s)` (run-3)

## 1. Summary

This change adds a stdlib-only `slugify(s) -> str` function under
`test-run-3/slugify.py` and an exhaustive pytest suite at
`test-run-3/test_slugify.py`, plus a one-line `.gitignore` addition for
`test-run-3/`. The implementation is 6 effective lines of code (within the
spec's "~12 lines" target), follows the spec's algorithm exactly (NFKD →
strip Mn → lower → `re.sub(r"[^a-z0-9]+", "-", ...)` → `strip("-")`),
raises `TypeError` with the spec's prescribed message format, and produces
the contractually required behaviors. Test output: 40 passed, 1 xfailed
(case 19, eszett); the xfail is the engineer's documented escalation of a
spec assumption that does not hold on this Python build (see Question
Q1). All 35 spec-mandated cases are encoded.

The implementation improves the codebase. There are no Blocking findings.
There is one Question for the architect (the eszett xfail decision) and a
small number of Important / Nit / Praise items.

## 2. Findings

### Important — eszett xfail vs. spec text §4 testing requirement #19

`test_slugify.py:72-82` marks case 19 (`"Straße"` → `"strasse"`) as
`xfail(strict=True)` with `condition = not _ESZETT_DECOMPOSES`, where
`_ESZETT_DECOMPOSES` is computed at import time as
`unicodedata.normalize("NFKD", "ß") == "ss"`. On the test environment
(CPython 3.13.12, conda-forge), I independently verified
`unicodedata.normalize("NFKD", "ß") == "ß"` (not `"ss"`), so the xfail
fires and the suite reports `40 passed, 1 xfailed`.

This is the correct behavior under the spec's escape hatch — §5 Open
Questions #6 explicitly anticipates the failure mode and says: "engineer
should report it rather than silently change the spec." The engineer
preserved the spec's expected value (`"strasse"`) rather than changing it
to whatever the implementation actually produces (`"stra-e"`), which keeps
the spec contract honest and makes the xfail self-healing on Python builds
where ß does decompose.

Why it matters: this is a deviation from a literal reading of §4 #19
("must pass") in favor of §5 #6 ("report rather than silently change").
The architect needs to confirm (Q1 below) that this resolution is
acceptable, since the literal §4 line "All 35 cases must pass" is no
longer strictly true — instead, 34 pass and 1 xfails by design.

Suggested resolution: architect confirms acceptance in Mode 3; if
accepted, optionally update spec §4 #19 in a future iteration to note
the xfail contingency, so future readers do not see a surprising xfail
without context. No code change required for this run unless the
architect rules otherwise.

### Important — `requirements.txt` absent (spec-permitted, but report needed)

The spec (§4 file-level structure, point 3) states `requirements.txt` is
"optional — engineer may omit if they prefer the lighter `test-run-2/`
style." The engineer chose to omit. `ls test-run-3/` shows only
`slugify.py` and `test_slugify.py` (plus `__pycache__`). This is within
spec, but spec also says: "State the choice in the completion report."

Why it matters: the reviewer cannot verify from spec + diff + test output
alone whether the engineer documented the choice in
`engineer-completion.md` (reviewer is isolated from that file per ADR
020 / Cato reviewer-isolation rule). Architect Mode 2 should already
have validated this; flagging here so the architect can confirm the
report covers it during Mode 3 triage. No code change required.

### Nit: docstring lost the diacritic example characters

`slugify.py:10-11` says `"cafe" -> "cafe"` and `"naive" -> "naive"`
where the spec docstring (§4 Interface) intends `"café" -> "cafe"` and
`"naïve" -> "naive"`. The intended diacritic characters appear to have
been ASCII-folded (likely during paste or editor handling), so the
docstring's example loses its illustrative value — both sides of the
arrow are identical ASCII strings, which makes the rule look like a
no-op. The implementation itself is unaffected; this is purely
documentation polish.

Suggested resolution: change to `"café" -> "cafe"` and `"naïve" -> "naive"`
(or restore whatever the editor displays as the intended source
characters). Author may ignore.

### Nit: case 5 is functionally identical to case 4

`test_slugify.py:39, 41`: case 4 (`single-ascii-word`) and case 5
(`already-clean-slug`) both encode `slugify("hello") == "hello"`. The
spec lists them as separate items (case 4 = "Single ASCII word", case 5
= "Already-lowercase, no separators (idempotency on a clean slug)"), so
the engineer is faithful to the spec. But functionally the second test
exercises the same code path as the first.

Why it doesn't matter much: the spec asked for both, the engineer
delivered both, and the cost is one parameter row. The redundancy is
the spec's, not the engineer's.

If the architect ever revises the spec, case 5 could be merged into
case 31 (idempotency) using `"hello"` as one of the inputs. No action
for this run.

### Nit: emoji and diacritic source characters use escape sequences vs. literals inconsistently

`test_slugify.py:96, 98` use `\U0001f44b` (escape form) for the waving-
hand emoji, while em-dash, en-dash, CJK characters, Cyrillic, and the
diacritics elsewhere are written as literals. Both are valid Python and
both round-trip identically, but the inconsistency is mildly distracting
when reading the test cases. Author may ignore — the escape form for
emoji has a defensible reason (some editors / fonts render emoji
inconsistently in source).

### Praise: faithful, minimal implementation

`slugify.py` is 6 lines of code (29 lines including docstring), exactly
the shape the spec asked for. No defensive over-engineering, no extra
keyword arguments, no apostrophe pre-pass, no manual ligature mapping
table — every "do not add X" warning in the spec is honored. The
TypeError uses the exact prescribed message format
(`f"slugify() expected str, got {type(s).__name__}"`).

### Praise: thoughtful eszett xfail mechanism

`test_slugify.py:25, 72-82` computes `_ESZETT_DECOMPOSES` at module load
time and uses it as the `xfail` condition. This means the test will
auto-pass (and the xfail will become an unexpected XPASS under
`strict=True`, surfacing the change) on any future Python build where
NFKD on ß starts producing `"ss"`. That is a self-healing pattern that
will tell future maintainers to remove the xfail rather than silently
masking a regression. Good defensive design.

### Praise: invariant tests (cases 31 and 32) are real assertions, not smoke tests

`test_slugify.py:131-132` actually computes `slugify(slugify(x))` and
compares against `slugify(x)` — not just "doesn't raise." Case 32
similarly uses a real anchored regex (`^[a-z0-9]+(-[a-z0-9]+)*$`) that
would fail on doubled hyphens, leading/trailing hyphens, or stray
characters. Both invariant tests will fail loudly on the obvious bugs
they target.

## 3. Questions

### Q1 — Eszett xfail: accept as the contract for this run?

Observed: `test_slugify.py` marks spec case 19 (`"Straße"` → `"strasse"`)
as `pytest.mark.xfail(strict=True)` when NFKD on ß does not produce
`"ss"`. On the test environment (Python 3.13.12, verified), it does not,
so the test xfails. Suite output: `40 passed, 1 xfailed`.

Unclear: spec §4 #19 says "must pass"; spec §5 Open Questions #6 says
"engineer should report it rather than silently change the spec." These
are in tension when the assumption underlying §4 #19 is false on the
target Python.

What I would do under each interpretation:

- If the architect rules "xfail is the right escalation for this
  spec-vs-environment conflict, accept the run as-is": no code change;
  triage other findings as Important/Nit; this can merge.
- If the architect rules "the spec's assumption was wrong; revise the
  spec to acknowledge the eszett dependency on Python build, and accept
  xfail going forward": no code change for this run; spec edit is a
  separate (out-of-scope) follow-up.
- If the architect rules "the spec wins; engineer must add a manual
  pre-pass mapping `ß → ss` (and possibly `Æ → ae`, `Œ → oe`) to make
  case 19 pass without xfail": this is a meaningful spec/code change.
  Spec §5 #5 explicitly allows this as an opt-in; spec §5 #6 does not
  request it. Architect call.

I treat this as Important rather than Blocking because the engineer
took the spec-prescribed escalation path (§5 #6: "report rather than
silently change"). If the architect prefers a different resolution, this
becomes a must-fix; otherwise it can merge as-is.
