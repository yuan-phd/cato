## 2. Findings

### Praise: `test-run-1/reverse.py:1-9`
The implementation is exactly the size the spec called for: one import, one type guard, one tokenization call, one join. No helpers, no caching, no logging, no `__all__`, no module-level constants. This is the right shape for a function whose entire job is one well-defined transformation.

### Praise: `test-run-1/reverse.py:7`
The `TypeError` message is constructed via `type(s).__name__` rather than `repr(s)` or `str(s)`, which keeps the message bounded and avoids leaking large inputs or invoking arbitrary `__repr__` methods on the rejected object. Matches the spec's literal format string verbatim.

### Praise: `test-run-1/test_reverse.py:99-107`
`test_reverse_no_nfc_mutation` is the most load-bearing assertion in the suite (it is the only one that distinguishes a correct implementation from an "almost-correct" one that silently normalizes), and it is well-written: it asserts both `result == nfd_input` *and* `result != nfc_input`, so a hypothetical implementation that collapsed input to NFC would fail the second assertion even though the visible glyph would still match. The comments naming the code-point sequences (U+0065 U+0301 vs. U+00E9) make the intent unambiguous to a future reader.

### Praise: `test-run-1/test_reverse.py:84-92`
Type-error tests assert on the *content* of the exception message (`"reverse() expected str, got {typename}"`), not just on the exception class. This satisfies the spec's "Verify exception type, not just that *something* is raised" concern stringently.

### Nit: `test-run-1/test_reverse.py:17,19,21,51,102-103`
The combining-acute test cases (7, 8, 9, 15, 20) and the `nfd_e_acute` literal in case 20 use raw NFD characters in the source file (e.g., the literal string `"é"` written as `e` + U+0301) rather than the explicit escape form `"é"` named in the spec note ("Use escape literals (`"́"`, `"\U0001D11E"`, `"‍"`) — not raw bytes"). The values are byte-correct — case 20 in particular passes precisely because it asserts the NFD/NFC distinction at runtime, which would not pass if the literals had been silently re-encoded. So this is purely a literal-form deviation, not a correctness deviation.

The trade-off:
- Cost: any tool that round-trips the file through an editor with auto-NFC, a different encoding, or a paste through a normalizing clipboard could silently turn the NFD literals into NFC and case 20 would start failing for a reason that is invisible in a diff.
- Benefit of the current form: the test reads more naturally to a human eye.

I rate this Nit rather than Important because (a) the file as committed is byte-correct and tests pass, (b) the test suite is workflow-validation throwaway code under a gitignored directory, and (c) case 20 itself is the canary — if NFC contamination ever occurred, that single test would catch it loudly. If this code had a longer life, I'd push for escape-literal form. As-is, author's choice.

Suggested resolution (optional): rewrite the affected literals to `"é"`, `"café"`, `"áb́"`, etc., to make the code-point structure explicit in source.

### Nit: `test-run-1/test_reverse.py:24,30,50,68,72`
ZWJ characters (U+200D) inside the test string literals are likewise written as raw zero-width characters rather than `"‍"` escapes. Same trade-off as above, same rating. The strings are invisible-glyph soup to anyone scrolling past — escapes would make the structure self-documenting. Not blocking.

### Nit: `test-run-1/requirements.txt:2`
The spec said `pytest` is dev-only and "omit or add as comment". The current comment `# pytest is dev-only, not listed here` is fine but slightly awkward; a more conventional form would be `# pytest  # dev-only, install separately` or simply omitting it. Trivial.
